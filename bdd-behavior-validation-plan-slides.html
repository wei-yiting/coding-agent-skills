什麼是 BDD？

"BDD is about using examples in conversation to illustrate behaviour."
— Liz Keogh

核心理念

  • Behavior-Driven Development 以具體範例作為溝通工具
  • Business、Dev、QA 三方對話來發現需求
  • Given-When-Then 語法轉化為可執行規格
  • 由 Dan North 在 2003 年從 TDD 演化而來

BDD 不是什麼

  • 不只是測試方法 — 是需求發現流程
  • 不是事後補的驗收測試 — 是 coding 前的行為規格
  • 不是工具（Cucumber）— 工具只是形式化手段

Dan North：用 should 取代 will — 「系統應該做 X」隱含了可被挑戰的空間

關鍵原則

  • Declarative — 描述 what，非 how
  • Self-contained — 每個 scenario 獨立
  • Business language — 非技術術語

Three Amigos 方法

George Dinwiddie (2009) — 三種認知空間的協作對話

👤

Product Owner

The One Who Requests

Value Space

「這有價值嗎？」

  • 提出初始具體範例
  • 判斷商業價值
  • 決定納入/排除/降級
⚙️

Developer

The One Who Suggests

Solution Space

「怎麼建構？暗示什麼約束？」

  • Mental sketch implementation
  • 從建構浮現約束
  • Counter-examples 表達
🔍

QA Tester

The One Who Protests

Problem Space

「這會怎麼壞掉？」

  • Boundary value analysis
  • Failure mode exploration
  • 找出隱藏假設

JFS："It is very hard to create and criticise at the same time." — 三角色 = 創造 + 請求 + 批判

為什麼用 Agent Teams 而非 Sub-agents？

Challenge Rounds — 多輪辯證

Completion Criteria

  • 每個 Rule ≥ 1 Dev + 1 QA challenge
  • 沒有換名字重述
  • ≥ 1 challenge 揭露 design 未定義行為
  • PO 已判決所有 challenge，contests 已解

Discovery Conversation 範例

Rule: Conversation persists within session

PO
Seeds：Alice 同 session 兩訊息 → 能參考。Bob 跨 session → 無法參考。
Dev
Challenge (Persistence)：Checkpointer 若 in-memory，server restart → session 消失。使用者只會看到 agent 忽略對話。
QA
Challenge (Boundary)：50 輪對話？LLM context window 丟掉早期 turns，checkpointer 還留著。靜默失敗。
Dev
Cross-talk：QA 的 50 輪場景揭露 context window divergence — checkpointer 有 turn 1，但 LLM 看不到。
PO
Judgment：50 輪 → Include Server restart → Needs input ⚡ QA contest（mobile 使用者）→ Include

Execution Phase 全流程

BDD E2E Verification Loop

實作完成後，自動化執行 verification-plan.md 中的所有場景

BDD E2E Loop 做什麼？

  • 讀取 verification-plan.md
  • 自動執行所有 Deterministic 驗證（curl/script）
  • 自動執行所有 Browser Automation 驗證
  • 失敗 → dispatch Fixer sub-agent 修復
  • 修復後重新跑驗證，直到全部通過

Manual Verification

  • Test Limitation — 技術上無法自動化的測試（需實體裝置、高併發環境等），由使用者手動執行
  • User Acceptance Test — 最終的 overall flow sanity check，由 PO 視角驗收

Manual tests 和 automated BDD verification 都通過後，才進入 Code Review。

Code Review Loop

兩個平行 Sub-agent 交互收斂，直到 0 review issues

為什麼用兩個不同模型？

角色模型原因
ReviewerCodex快速、大量 code reading
FixerOpus 4.6深度推理、精確修復

Review 完成後

  • 再跑一次 BDD Verification Plan — 確認 code review 的修改沒有 regression
  • 同時確認 Unit Tests、Type Check、Lint 全部通過
  • 全部通過 → 可以發 PR

PR + Final User Acceptance

  • PR 發出後，使用者做最後的 User Acceptance Test
  • 這是 verification-plan.md 中 Manual UAT 的最後一步
  • UAT 通過 → Merge

Summary

Part 1 — Planning

  • BDD 用具體範例驅動需求發現
  • Three Amigos 三種認知空間（Value / Solution / Problem）辯證
  • Agent Teams（非 Sub-agents）讓 AI agents 直接互相挑戰
  • Challenge Rounds 多輪結構化辯證，含 cross-talk 和 contest
  • 產出 bdd-scenarios.md + verification-plan.md

Part 2 — Execution & Validation

  • Sub-agent Driven Dev 用 TDD 實作 implementation.md
  • BDD E2E Loop 自動驗證 + Fixer 修復循環
  • Manual Verification Test Limitation + User Acceptance Test
  • Code Review Loop Reviewer (Codex) + Fixer (Opus 4.6) 交互收斂
  • Final BDD + PR regression check → PR → UAT → Merge

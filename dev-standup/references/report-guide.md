# Standup Report Guide — content spec & HTML patterns

The template (`assets/template.html`) is htmlify's template plus a dashboard layer
(`STANDUP DASHBOARD` CSS block + collapsible comment panel). All htmlify machinery works
unchanged: sidebar TOC, mermaid lightbox, comment system (text-selection → sidecar JSON),
light/dark. You only fill placeholders and emit content HTML using the classes below.

Template invariants — never remove when editing the template: the **comment panel is
collapsible**（header 的 ⇥ 收合、右緣浮動 💬 展開、狀態存 localStorage、新增 comment 自動展開）,
and prose inside cards is capped to a readable measure（`.issue-card > p`、`.next-step`
≤ 48rem — 行寬太長會毀掉中文閱讀性）. Both exist because of explicit user feedback.

## Why these sections exist (read before writing)

The report must be **complementary to Linear, not a text mirror of it**. Linear already
shows titles, statuses, and comments. What Linear *cannot* show — and what this report is
for — is:

1. **Synthesis across sources**: Linear intent × git worktree reality × PR state, cross-checked.
2. **Prioritization with reasons**: not "here are 10 issues" but "these 3 matter today, because…".
3. **The dependency picture**: what blocks what, which single action unblocks the most.
4. **Ball ownership**: for every issue, is the next move the human's or the agent's — and
   for human moves, how urgent.
5. **Plain-language narrative**: someone who forgot everything should re-load the whole
   picture from this page alone, without clicking into Linear.

If a sentence in the report could have been copy-pasted from Linear, it's not earning its
place. Every issue narrative must answer: 這是什麼、為什麼重要、現在卡在哪、下一步是誰的球。

## Writing rules for narratives

- **No bare IDs.** Never write "DONG-63 的 P0-1" as the whole explanation. Write what the
  thing actually is: "eval loader 把 CSV list 欄位當字串處理，導致 sec_retrieval 指標全是雜訊"。
  IDs appear *after* the description, as anchors — not instead of it.
- **2–4 sentences per issue card narrative**, in 台灣繁中, technical nouns in English.
- **State the consequence, not just the state.** "dirty:27" is a state; "27 個未 commit 檔案
  只存在這台機器上，包含已通過驗證、只差 commit 的成果" is a consequence.
- **Next step must be imperative and concrete**: who does it, what exactly, and the command
  if one exists.
- **Every issue id is a link to Linear.** Build the URL from the `url` field `list_issues`
  returns (short form `https://linear.app/<workspace>/issue/DONG-XX` redirects fine):
  `<a class="issue-link" href="…" target="_blank">DONG-60</a>`. Apply everywhere —
  section `<h2>` titles, queue items, prose, focus list — **except** inside mermaid node
  labels, inside `resume-cmd` code, and sidebar TOC labels (nested `<a>` is invalid). The
  easiest way is to write plain `DONG-XX` everywhere and run one linkify pass at the end
  (regex over the `<main>` region, skipping mermaid blocks / `resume-cmd` / `toc-label`
  lines).

## Placeholder mapping

| Placeholder | Fill with |
|---|---|
| `{{DOC_TITLE}}` | `Dev Standup — YYYY-MM-DD`（focused mode: `Standup — DONG-63 — YYYY-MM-DD`）|
| `{{DOC_SUBTITLE}}` | one-line state of the world, e.g. "10 issues in flight · 2 human decisions blocking · 37 dirty files at risk" |
| `{{STATUS_BADGE}}` | omit |
| `{{HERO_STATS}}` | `stats-grid` with 4 stats: in-flight issues / human actions pending / agent-ready items / at-risk file count |
| `{{HERO_META}}` | omit, or one `.prereq` callout if a single fact dominates the day (e.g. "所有 P1 都卡在 DONG-64 的決策") |
| `{{SIDEBAR_BRAND}}` | `Dev Standup` |
| `{{SIDEBAR_SUB}}` | the date |
| `{{SIDEBAR_TOC}}` | one `<li>` per section (see skeleton) |
| `{{LEARNING_PANEL}}` | leave untouched (no panel) |
| `{{MAIN_CONTENT}}` | the sections below |
| `{{FOOTER_TEXT}}` | `Generated by dev-standup · YYYY-MM-DD HH:MM` |
| `{{STORAGE_KEY}}` | `standup-comments-<YYYY-MM-DD>-v1`（focused: `standup-comments-dong-63-<YYYY-MM-DD>-v1`）|
| `{{DOC_NAME}}` | same as `{{DOC_TITLE}}` |

Section ids are sequential `s1, s2, …` in document order (the TOC and comment system key
off them). Issue cards are sections too — each gets its own `sN` and TOC entry, so
comments anchor to the specific issue.

## Section skeleton (full-standup mode)

| # | Section | Content |
|---|---|---|
| s1 | 🎯 今日焦點 | top 3–5 prioritized items, `.focus-list` |
| s2 | 🧭 行動佇列 | `.queue-cols`: human queue (by urgency) + agent-ready queue |
| s3 | 🗺️ Dependency 地圖 | mermaid graph + 2–3 sentence reading of it |
| s4…sN | issue cards | one section per in-flight issue, `.issue-card` |
| sN+1 | ⚠️ 風險與追蹤缺口 | lone copies, NO-UPSTREAM, tracking gaps |
| sN+2 | Δ 自上次 standup | only if a previous `state.json` exists |

Focused mode (single issue/worktree) replaces s1–s3 with: s1 = the issue's full narrative
(background, decisions so far, why it matters), s2 = current state deep-dive (worktree,
diff summary, artifacts, PR state, review findings), s3 = 下一步 checklist split
human/agent, s4 = its dependency neighborhood (what it blocks / is blocked by).

## HTML patterns

### s1 今日焦點

Order by the prioritization rubric in SKILL.md. Each item: what + why-now.

```html
<section id="s1" class="doc-section">
  <div class="section-head">
    <span class="section-num">§1</span>
    <h2>🎯 今日焦點</h2>
    <span class="section-tag tag-decision">Priority</span>
  </div>
  <ol class="focus-list">
    <li>
      <div>
        <span class="badge badge-human">HUMAN</span>
        <span class="q-title">回覆 repo 稽核的 6 個決策點（DONG-64）</span>
        <div class="focus-why">純人工決策、零程式工作，但它卡住 eval 修復的 E1/E3、
        5-batch plan、和 Finnhub PR 的 merge 策略 — 回覆後三條線同時解鎖。</div>
      </div>
    </li>
  </ol>
</section>
```

### s2 行動佇列

```html
<section id="s2" class="doc-section">
  <div class="section-head">
    <span class="section-num">§2</span>
    <h2>🧭 行動佇列</h2>
  </div>
  <div class="queue-cols">
    <div class="queue queue-human">
      <h3>🧑 需要你（按急迫度）</h3>
      <div class="queue-item">
        <span class="badge badge-p0">P0</span>
        <span class="q-title">決策點 3 &amp; 6 — language-policy 收斂、eval 單跑</span>
        <div class="q-why">DONG-63 的 session 今早已開工，E1/E3 兩項修法完全取決於此。</div>
      </div>
    </div>
    <div class="queue queue-agent">
      <h3>🤖 Agent 隨時可做</h3>
      <div class="queue-item">
        <span class="badge badge-agent">READY</span>
        <span class="q-title">quant-yfinance：commit 已驗證的 404 fix → 開 PR</span>
        <div class="q-why">審查已放行（187 tests passed），純機械收割。</div>
        <code class="resume-cmd">cd …/quant-yfinance-ingestion && claude "執行 DONG-61 收尾"</code>
      </div>
    </div>
  </div>
</section>
```

Queue membership rule: an item goes in the human queue only if the *next* move is a
decision, review, credential, or approval only the human can produce. Everything already
unblocked and mechanical goes in the agent queue with a ready-to-paste command.

### s3 Dependency 地圖

Build edges from Linear blocker relations **and** blockers stated in issue descriptions
("⛔ Blocked by", "前置條件"). Style nodes by state; keep labels short (id + 3–5 字摘要).
Read the graph for the user below it — the single highest-leverage node, chains about to
unblock.

```html
<section id="s3" class="doc-section">
  <div class="section-head">
    <span class="section-num">§3</span><h2>🗺️ Dependency 地圖</h2>
    <span class="section-tag tag-architecture">Map</span>
  </div>
  <div class="mermaid">
graph LR
  D64["DONG-64<br/>稽核決策"]:::human --> D63["DONG-63<br/>eval 修復"]:::active
  D60["DONG-60<br/>streaming PR"]:::harvest --> D75["DONG-75<br/>frontend"]:::backlog
  D60 --> D76["DONG-76<br/>docs 重寫"]:::backlog
  D61["DONG-61<br/>quant PR"]:::harvest --> D74["DONG-74<br/>config 收斂"]:::backlog
  D63 --> D62["DONG-62<br/>eval PR"]:::active
  classDef human fill:#fbeeda,stroke:#b4640f,color:#9a5410
  classDef active fill:#d6ecee,stroke:#0c7d84,color:#0c7d84
  classDef harvest fill:#e2f4ea,stroke:#147045,color:#147045
  classDef backlog fill:#f0f2f5,stroke:#c6ccd4,color:#6b7480
  </div>
  <p>讀圖：… 哪個節點解鎖最多、哪條鏈快通了 …</p>
</section>
```

classDef 語意：`human` = 等人決策（橘）、`active` = 進行中（teal）、`harvest` = 只差收割（綠）、
`backlog` = 排隊中（灰）。圖例可加一行說明。

### Issue card (s4…sN)

One `<section>` per in-flight issue. Title row = §N + issue id + title + badges.

```html
<section id="s4" class="doc-section">
  <div class="section-head">
    <span class="section-num">§4</span>
    <h2>DONG-63 — eval pipeline 修復</h2>
    <span class="badge badge-agent">AGENT 進行中</span>
    <span class="badge badge-stage">Implementation</span>
    <span class="badge badge-block">部分卡決策</span>
  </div>
  <div class="issue-card">
    <p><strong>這是什麼</strong>：… 2–4 句 narrative（見 writing rules）…</p>
    <p><strong>目前進度</strong>：… 上次 sync 說了什麼、實際 git 狀態是什麼 …</p>
    <div class="card-meta">
      <span class="meta-chip">branch fix/eval-pipeline-sec-retrieval</span>
      <span class="meta-chip danger">NO-UPSTREAM</span>
      <span class="meta-chip warn">dirty 7</span>
      <span class="meta-chip">last commit 5w ago</span>
    </div>
    <div class="next-step"><strong>下一步（agent）</strong>：續跑今早的 session，完成不需決策的三項。</div>
    <code class="resume-cmd">cd …/eval-pipeline-fixes && claude --resume a6c0b85b-…</code>
  </div>
</section>
```

Meta-chip severity: `danger` = data-loss risk (NO-UPSTREAM, unpushed commits),
`warn` = dirty files / stale, `ok` = pushed & clean. Show a `.resume-cmd` whenever a
resume command exists in the latest sync comment.

### ⚠️ 風險 & Δ sections

Risk section: `.callout-warn` per violation of the no-lone-copies rule, each with the
concrete fix action. Delta section:

```html
<ul class="delta-list">
  <li><span class="delta-tag done">DONE</span> DONG-58 救援完成 — 三個 local-only branch 都已 push</li>
  <li><span class="delta-tag new">NEW</span> DONG-70~76 新開 7 張稽核修復 issues</li>
  <li><span class="delta-tag moved">MOVED</span> DONG-63 Backlog → Implementation（session 今早開工）</li>
  <li><span class="delta-tag risk">RISK</span> multi-provider worktree dirty 從 12 → 27</li>
</ul>
```

## state.json — the delta snapshot

After writing the HTML, save `standup/state/<YYYY-MM-DD>.json` next to the reports. Next
standup diffs against the most recent one. Keep it small and stable:

```json
{
  "generated_at": "2026-07-07T10:30:00+08:00",
  "mode": "full",
  "issues": {
    "DONG-63": { "status": "Implementation", "owner_next": "agent", "priority_rank": 3 }
  },
  "worktrees": {
    "eval-pipeline-fixes": { "branch": "fix/eval-pipeline-sec-retrieval", "dirty": 7, "unpushed": "NO-UPSTREAM(0)", "last_commit": "5 weeks ago" }
  },
  "risks": ["multi-provider dirty:27", "v3-quant-research no Linear issue"]
}
```

Delta = compare `issues` (new / completed / status moved / owner flipped), `worktrees`
(dirty/unpushed deltas), `risks` (appeared / cleared). Don't diff prose.

## Assembly mechanics

The template is ~2900 lines — don't Read it whole. Copy it to the output path, then:
grep for `{{` to get placeholder line numbers, Read each placeholder's surrounding lines,
Edit them in place. `{{MAIN_CONTENT}}` is one marker replaced with all sections in a
single Edit. After filling, grep `{{` to confirm nothing is left, then `open` the file.
